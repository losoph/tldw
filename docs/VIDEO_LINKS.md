# 🎬 Парсинг видео по ссылке (VK / YouTube) — план и варианты

Заметки по загрузке «по ссылке». Задача разбита на два контура — **VK Видео** и
**YouTube (Google)** — потому что ломаются они по разным причинам и лечатся разными
средствами. Здесь фиксируем: что уже сделано в коде, почему могло не работать вообще
(ни приватное, ни публичное), и какие есть варианты дальше.

---

## 1. Текущая реализация (worker.py → `download_url`)

Скачиваем **только аудиодорожку** целиком (`-f bestaudio/best`), фрагменты режем уже
локально ffmpeg'ом (см. `_process`). Раньше клипы вырезались на этапе скачивания через
`--download-sections` — от этого отказались: с несколькими фрагментами таймкоды сбивались,
а сама опция на VK работала нестабильно.

### Каскад попыток (главное исправление доступа)

Ключевая находка: **протухшие / привязанные к IP cookies ломают ДАЖЕ публичные видео** —
VK и YouTube отвергают битую сессию и отвечают «only available for registered users» /
«Sign in to confirm you're not a bot», хотя без cookies ролик скачался бы анонимно.
Раньше код всегда подставлял cookies, если файл присутствовал, — и público-загрузка падала.

Теперь `download_url` пробует по очереди, пока не получится:

| # | VK | YouTube |
|---|----|---------|
| 1 | с cookies (если есть) | `player_client=default` + cookies (если есть) |
| 2 | **без cookies** (публичный путь) | `player_client=default` без cookies |
| 3 | — | `player_client=android` без cookies |
| 4 | — | `player_client=tv` без cookies |

Разные `player_client` — штатный обход антибот-защиты YouTube. Между попытками чистим
недокачанные файлы. Если всё упало — отдаём понятную ошибку (`friendly_download_error`),
которая различает «нужна авторизация» и «устарел yt-dlp / антибот».

### Прочее в коде
- `--retries 3`, `--socket-timeout 30` — устойчивость к сетевым сбоям.
- Прокси per-domain из `data/proxies.yml` (VK — напрямую, YouTube — через прокси).
- deno-рантайм из `data/bin/deno` для n-challenge YouTube.
- Временные файлы (исходник, WAV, `.title`, `.part`) удаляются в `finally` — и при ошибке.

### Что проверить в первую очередь, если «не качает вообще»
1. **Версия yt-dlp** — самая частая причина. VK/YouTube ломают старые версии почти
   каждые пару недель. Обновить: `bash scripts/update-ytdlp.sh` (или `tldw rebuild`).
   Проверить: `docker compose exec worker yt-dlp --version`.
2. **Убрать/переименовать битые cookies** и проверить публичный ролик — если заработало,
   проблема была в cookies (см. § про IP-привязку в OPERATIONS.md).
3. **Ручной прогон в контейнере** для точной диагностики:
   ```bash
   docker compose exec worker yt-dlp -f bestaudio/best --no-playlist -v "<URL>"
   ```
4. **Прокси YouTube** — с российского IP YouTube нестабилен; проверить `data/proxies.yml`.

## 2. VK Видео — план

**Публичные ролики** должны качаться `yt-dlp` без cookies (каскад, попытка #2). Если не
качаются даже публичные — почти всегда **устаревший yt-dlp** или сменившийся VK-плеер.

**Приватные / только для авторизованных** — нужны cookies корпоративного аккаунта,
привязанные к IP сервера (SSH-туннель, инструкция в OPERATIONS.md → «Обновление VK-cookies»).

Варианты на будущее (если yt-dlp продолжит ломаться):
- **VK API (video.get)** — официальный, стабильный. Нужен `access_token` приложения с правом
  `video`. Отдаёт прямые ссылки на mp4 разных качеств. Плюс: не зависит от плеера/челленджей.
  Минус: токен и права, приватные видео — только с токеном пользователя, у кого есть доступ.
  → Кандидат №1 для надёжности VK. Прототип: `video.get` по `owner_id`/`video_id` из URL.
- **Собственный извлекатель `m3u8`** из HTML страницы плеера (VK отдаёт HLS-мастер). Хрупко,
  ломается при редизайне плеера — только как крайний фолбэк.
- **gallery-dl** — альтернативный загрузчик, иногда переживает поломки, которые ломают yt-dlp.

## 3. YouTube (Google) — план

Основная боль — антибот («Sign in to confirm you're not a bot») и n-challenge (нужен JS-рантайм).

Уже сделано: перебор `player_client` (default/android/tv), deno-рантайм, прокси. Дальше:
- **Пул `player_client`** держать в актуальном состоянии: значения (`android`, `ios`, `tv`,
  `web_safari`, `mweb`) yt-dlp периодически меняет — сверяться с релиз-ноутами yt-dlp.
- **cookies одноразового Google-аккаунта**, залогиненного через тот же прокси, что качает
  сервер (IP должен совпадать) — снимает антибот для большинства роликов.
- **PO Token** (`--extractor-args youtube:po_token=...`) — Google всё чаще требует Proof-of-Origin
  токен. Его генерация — отдельная история (bgutil-провайдер / внешний сервис). Заложить,
  когда антибот начнёт требовать массово.
- **YouTube Data API v3** — метаданные отдаёт, но **не** прямые ссылки на поток (легально скачать
  файл нельзя). Для нашей задачи (нужна аудиодорожка) API не заменяет yt-dlp.
- **Резидентный / зарубежный прокси** — датацентровые IP Google быстро метит. Резидентный прокси
  или собственный VPS за рубежом резко повышают проходимость. См. открытый вопрос в TLDW_CONTEXT §7.7.

## 4. Общий фолбэк-инструмент (обоих контуров)

- **cobalt** (github.com/imputnet/cobalt) — self-hosted сервис-загрузчик с HTTP API, поддерживает
  и YouTube, и VK. Можно поднять рядом в docker-compose и ходить в него, если yt-dlp упал.
  Кандидат в «второй движок» для устойчивости. Оценить после стабилизации yt-dlp-пути.
- **Единый интерфейс загрузчика** в worker.py: обернуть скачивание в абстракцию
  `fetch_audio(url) -> path` с несколькими бэкендами (yt-dlp → cobalt → VK API), перебор по
  аналогии с текущим каскадом попыток. Рефактор на потом, когда будет ≥2 рабочих бэкенда.
