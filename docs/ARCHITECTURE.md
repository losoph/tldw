## Поток обработки

1. **Загрузка** — пользователь выбирает файл, браузер отправляет multipart-POST на `/upload`
2. **Приём** — nginx буферизует и проксирует на FastAPI (`web`)
3. **Сохранение** — `web` сохраняет файл в `/data/uploads/{uuid}.{ext}`, создаёт запись в SQLite со статусом `pending`
4. **Поллинг** — каждые 3 секунды `worker` смотрит в БД на задачи в статусе `pending`
5. **FFmpeg** — извлечение аудио в WAV 16kHz mono → статус `extracting_audio`
6. **ffprobe** — замер длительности, расчёт оценки времени
7. **Whisper** — транскрибация локально → статус `transcribing`
8. **DeepSeek** — отправка текста в API, получение структурированного саммари → статус `summarizing`
9. **Завершение** — сохранение саммари в БД, удаление временных файлов → статус `done`
10. **Отображение** — фронтенд поллит `/status/{id}` каждые 1.5 сек и обновляет UI

## Структура данных

### SQLite таблица `jobs`

| Поле | Тип | Описание |
|---|---|---|
| `id` | TEXT PK | UUID задачи |
| `filename` | TEXT | оригинальное имя файла |
| `status` | TEXT | pending / extracting_audio / transcribing / summarizing / done / error |
| `created_at` | REAL | unix timestamp создания |
| `summary` | TEXT | финальное саммари от DeepSeek |
| `error` | TEXT | сообщение об ошибке если status='error' |
| `audio_duration` | REAL | длительность аудио в секундах |
| `started_at` | REAL | unix timestamp начала обработки |
| `estimated_seconds` | INTEGER | оценка времени = audio_duration × RTF |
| `transcript` | TEXT | сырой транскрипт (не используется в UI) |

### Файловая система (volume `data/`)

```
data/
├── jobs.db           # SQLite база
├── uploads/          # временные исходные файлы (удаляются после обработки)
└── audio/            # временные WAV для Whisper (удаляются после обработки)
```

### Volume `whisper_cache`

Хранит скачанные модели faster-whisper в `/root/.cache/huggingface/`. Сохраняется между перезапусками контейнеров — модель скачивается один раз.

## Безопасность

- **nginx** на входе: `auth_basic` блокирует анонимный доступ. Файл `.htpasswd`
  генерируется entrypoint-скриптом контейнера из `WEB_LOGIN`/`WEB_PASSWORD` (`.env`)
  при каждом старте — секрет не хранится в репозитории и не нужен на хосте
- **`server_tokens off`** скрывает версию nginx
- **fail2ban** на SSH (если настроен) банит IP после 4 попыток
- **Docker network isolation**: порт `8000` (FastAPI) виден только внутри docker network, наружу торчит только nginx :80
- **healthcheck**: контейнер `web` периодически проверяется, при падении docker автоматически рестартит
- **`restart: unless-stopped`** на всех контейнерах: после reboot VPS поднимаются автоматически

## Ограничения текущей реализации

- **Один worker** — задачи обрабатываются последовательно. При нескольких пользователях очередь может расти.
- **SQLite-поллинг** — не подходит для распределённой обработки. Для масштабирования нужен Celery + Redis.
- **HTTP без TLS** — пароль передаётся в base64. Для продакшна нужен HTTPS через Let's Encrypt (требует домен).
- **CPU-only** — для GPU нужно изменить `device="cuda"` в worker.py и базовый образ Docker.

## RTF и оценка времени

Real-Time Factor — отношение времени обработки к длительности аудио.

| Конфигурация | RTF (примерно) |
|---|---|
| small + beam=2 + VAD на 2-core CPU | ~2.0 |
| medium + beam=2 + VAD на 2-core CPU | ~4.5 |
| small на 4-core CPU | ~1.2 |
| any на GPU | ~0.1 |

После каждой задачи воркер логирует фактический RTF — используйте его для калибровки `RTF` в `.env`.
